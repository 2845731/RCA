# 一、先用大白话讲清楚:这个项目到底在干什么

**根因定位(Root Cause Analysis)是什么?**
一套微服务系统(很多台服务器、数据库、缓存、网关协同工作)出故障时,往往**几十个组件同时报异常**——CPU高、内存满、响应慢。但真正的"罪魁祸首"通常只有一个或几个,其它都是被它连累的。这个项目要回答三件事:

1. **谁是真凶**(root cause component,根因组件)
2. **什么时候开始出事的**(occurrence time,故障发生时间)
3. **为什么出事**(reason,故障原因,比如"内存不足""网络丢包")

**多智能体(multi-agent)是什么?**
把上面这件复杂的事,拆成几个**小角色(Agent,智能体)**,每个角色只干一件小事,干完把结果传给下一个角色。这个项目里有 7 个角色,排成一条流水线:

| 角色(Agent) | 它负责的一件事(大白话) |
|---|---|
| DataAgent(数据角色) | 把原始监控数据读进来,算好"正常值的标准线" |
| AssociationAgent(关联角色) | 找出"哪些组件现在不正常" |
| FaultIdentificationAgent(筛选角色) | 从不正常的组件里,挑出几个"最可疑的嫌疑人" |
| CausalGraphAgent(因果图角色) | 画一张"谁会影响谁"的关系图 |
| LogTopologyAgent(日志拓扑角色) | 从日志里补充"谁调用了谁"的关系 |
| InterventionAgent(干预角色) | 给每个嫌疑人打分,排出"最像真凶"的名次 |
| CounterfactualAgent(反事实角色) | 做最终裁决,定下真凶、时间、原因 |
| EvaluationAgent(评估角色) | 有标准答案时,诊断"这次错在哪一步"(只用于调试) |

理解了这条流水线,下面的分析就好懂了。

**13 个样本只得了约 2.5 分(≈19%)。** 但比"分数低"更要命的,是下面这**五个反复出现的现象**。

## 现象 1:所有分数都挤成一团,失去了区分能力

模型内部给每个嫌疑人算了好几种分数。我把 13 个样本的分数都翻出来,发现:

- **置信度分(top1_confidence,系统对自己最高名次的把握)**:13 个样本全在 **0.506~0.583** 之间;
- **反事实分(CounterfactualScore)**:全都 ≈ **0.50**(0.48~0.59);
- **原因分(ReasonScore)**:只在 **0.665、0.685、0.71、0.715** 四个值里打转;
- **最终分(FinalScore)**:全在 **0.62~0.71**。

**大白话:** 不管是哪个样本、哪个嫌疑人,算出来的分都差不多,都卡在 0.5~0.7 这个窄区间里。这就像一场考试**所有人都打 60 分**——分数完全失去了"谁好谁坏"的区分作用。系统看起来在"打分排序",其实分数里**几乎没装进任何关于这次故障的真实信息**。

> 我前面用过一个词叫 **"打分不可辨识"**,翻译成大白话就是:**"换个样本,分数也不变;换个嫌疑人,分数也不变。分数和真相脱钩了。"**

## 现象 2:系统老是指认 MG01 / MG02 这两个网关(13 次里指了 8 次)

真凶其实五花八门(Mysql、Redis、Tomcat、apache、MG01 都当过真凶),但模型的**最终裁决 13 次里有 8 次是 MG01 或 MG02**(占 62%)。唯一一次组件答对(样本10)还是因为真凶**正好就是 MG02**,属于撞对的,不是分析对的。

> 我前面叫它 **"枢纽吸引子"**,大白话:**MG01、MG02 是整个系统里"和别人连线最多"的中间件网关;系统的打分机制偏偏奖励"连线多"的节点,所以不管真凶是谁,系统都爱往这俩身上推。**

## 现象 3:原因几乎永远答"内存不足"(13 次里答了 10 次)

标准答案的原因很多样:网络延迟、网络丢包、磁盘IO、内存。但模型 **10 次都答"high memory usage(内存不足)"**——哪怕真答案是"网络延迟"(样本5)、"网络丢包"(样本10),它也照说"内存不足"。

> 大白话:**原因这一项基本是个"复读机",不管真相是什么都念"内存不足"。** 后面会讲为什么。

## 现象 4:时间老是差几分钟,而且忽早忽晚没规律

四个问时间的样本:样本1 早了16分钟、样本4 晚了7分钟、样本6 早了14分钟、样本12 晚了4分钟。**误差方向乱七八糟,四个全错。** 说明系统**根本没有一个靠谱的"找故障起点"的办法**,纯靠瞎蒙。

## 现象 5:连"自我诊断"都是错的

系统里那个 EvaluationAgent 本来是用来告诉开发者"这次错在哪一步"的。结果 13 次里有 **5 次它报"TOOL-ERROR / 未知错误"**,包括一些明明是"排序排错了"的样本也被它归成"未知"。**连用来找 bug 的工具自己都坏了**,导致开发者根本不知道流水线到底烂在哪。

# 三、为什么会这样:一层一层拆开讲错误原因

下面按流水线顺序,**每个角色先讲它怎么工作(大白话),再讲它哪里坏了、为什么坏**,所有结论都有代码或日志证据。

## 3.1 DataAgent + AssociationAgent:"判断谁不正常"这一步就坏了

### 它怎么工作的
- DataAgent 读一整天的监控数据,对每条指标算一条**"正常上限"**:取这条指标一整天数值的**第 95 百分位**(把一天的数值从小到大排,排在 95% 位置的那个值)当上限,叫 **P95 阈值**。
- AssociationAgent 看故障那半小时里,哪些指标**超过了上限**。超得越多,叫**偏差(deviation)越大**;偏差大的组件就被判定"不正常"。

### 坏在哪(四个连环坑)

**坑 1:几乎所有组件都被判"不正常"(饱和)。**
日志里 14 个候选组件**全部**被判异常,分数挤在 0.53~0.78。这叫**"饱和"**——大白话就是**"警报器太灵敏,人人都在响,等于没响"**。一旦人人都不正常,这一步就**没法区分谁是真凶**,后面全乱。

**坑 2:把"根本不是故障的信号"也当成了故障。**
被判异常的指标里,有两类纯属冤枉:
- **配置常量**:像 `HeapMemoryMax`(JVM 最大内存上限)、`JVMMaxMemory`,这些是**设定好就不变的死数**,根本不会"异常",却被判成强异常。
- **全机房一起波动的背景信号**:像 `CacheMem`(缓存内存)、`MEMFreeMem`(空闲内存),**每一台机器上都在亮**。这叫**"共模信号"**——

  > **共模(common-mode)**,大白话:**整个机房一起涨、一起跌的背景波动**。好比全班同学这次考试都考差了,那是卷子难,不能说明某个同学有问题。这种"大家一起动"的信号,不是某一台的故障证据,但系统把它当成了证据。

**坑 3(代码层根源):为什么配置常量会被判异常?**
在 `telemetry.py` 里,判断"超标多少"要除以一个叫 **scale(尺度)** 的数:`scale = max(波动幅度, 标准差, ..., 中位数×0.05, 0.000001)`。对一条**几乎不变的常量**,前面那些波动项都≈0,于是 scale 变成一个**极小的数**。除以一个极小的数,结果就会**爆炸式变大**,于是常量被误判成"严重超标"。这是个纯机制 bug。

**坑 4(最关键、最隐蔽):真正有用的"独家警报"反而被抹平了。**
AssociationAgent 算分时,用了一个"相对偏差"的归一化:**某指标的偏差 ÷ 所有组件这条指标偏差的中位数**。问题来了——

样本里 Mysql02 有个指标叫 `Aborted Clients`(被拒绝/中断的客户端连接数),偏差高达 **5.6**(别的指标普遍才 1.0),这本来是**数据库出事的强烈信号**。但因为**只有 Mysql02 这一家有这个指标**,"所有组件的中位数"就等于它自己,5.6 ÷ 5.6 = **1.0**,瞬间和满地的 1.0 一样平庸了。

> 大白话:**只有数据库才有的关键报警(独家线索),因为"就它一家有",被归一化算法当成"大家都这样"给抹平了。最该被听见的警报,恰恰在最关键的时候被静音了。**

## 3.2 CausalGraphAgent:画"谁影响谁"的关系图,这一步是整个项目的核心病根

### 它怎么工作的
- 先从 trace(调用链路记录)统计"谁调用了谁",画出**调用图**(比如 Tomcat 调用 MG02)。
- 然后**把箭头反过来**,变成"故障传播图":如果 Tomcat 调用 MG02,那 MG02 出问题会连累 Tomcat,所以画 **MG02 → Tomcat**。
- 给每条边算一个**权重(edge weight,这条影响关系有多强)**,用三个因素加权:时间先后、相关性、组件类型先验。

### 坏在哪(五个坑,前两个是致命的)

**坑 1(致命):数据库和缓存在这张图里是"孤岛",系统看不见它们。**
trace 调用记录里**根本没有数据库和缓存的身影**(MySQL、Redis 不出现在调用链 span 里)。本来有个 LogTopologyAgent 专门想从日志里补出"服务→数据库"的连线,但它的四种策略(DBCP2、Jedis、JDBC、连接错误日志)在这个数据集上**全部命中 0 条**(日志明明白白写着"命中 0 条")。

结果:**Mysql01、Mysql02、Redis01、Redis02 在因果图里全是没有任何连线的孤立点。**

> 大白话:**真凶(数据库)站在图的角落里,和谁都没连线。而后面给嫌疑人打分,靠的就是"你能通过连线影响多少别的组件"。数据库一条线都没有,自然"影响不了任何人",于是永远被判定"不像真凶"。系统从结构上就对数据库这类真凶失明了。**

**坑 2(致命,也是"老指认 MG02"的根源):打分奖励的是"连线多",不是"是真凶"。**
InterventionAgent 的核心分叫 **ExplainScore(解释分)**:衡量"这个嫌疑人能顺着连线解释掉多少别人的异常"。

- MG01、MG02 是网关,**出去的连线最多**,能"解释"一大片 Tomcat 的异常 → 解释分最高 → 永远冠军;
- 真凶数据库孤立无连线 → 解释分≈0 → 永远垫底。

> 大白话:**这套打分本质上是在奖励"朋友圈最大"的节点,而不是"真正闯祸"的节点。MG01/MG02 朋友圈最大,所以总当选;数据库是个独行侠,所以总落选。这就是为什么 13 次里 8 次都指认 MG01/MG02。**

而第 0 部分那个 Mysql02 的样本更说明问题:Mysql02 在筛选角色那步**本来排第 1**(综合证据 0.7292 全场最高),结果一进因果图打分,被从 #1 一路压到 #3,让 MG02 上位。

> 我前面说的 **"因果机制净负收益"**,大白话就是:**本来不画这张图、直接挑"证据最强"的组件(Mysql02)还能猜对;结果加上这套因果图分析,反而把对的答案排到后面、把错的捧上去——越分析越错,这套分析是帮倒忙的。**

**坑 3:边的权重最后都变成了几个固定常数,没装进真实信息。**
边权重靠三个因素:
- **时间先后**:本该"原因先出事、结果后出事"。但因为前面"饱和"导致几乎所有组件的异常都被算成"从故障窗口一开始就有",大家起点一样,时间先后差≈0 → 这个因素直接失效(代码里返回 0.00001)。
- **相关性**:全员都在异常,曲线长得都像,相关性要么虚高要么算不出。
- **类型先验**:一张写死的表(比如"数据库→服务=0.55")。

三个里前两个失效,只剩第三个固定的表。于是日志里大量边权是 **0.1575、0.1654、0.2021** 这种**反复出现的固定值**(0.1575 = 0.35×0.45,纯粹是那张表的数)。

> 大白话:**这张"因果图"的边权,最后基本等于一张事先写死的固定表,跟这次故障的真实数据几乎没关系。** 这也解释了现象1"分数全挤成一团"——因为大家都在用同一张固定表。

**坑 4:箭头方向被一刀切地写死,还画出了"环"。**
代码一律认为"被调用方出事 → 连累调用方"。但现实里**两个服务可能互相影响**(README 自己都承认 Tomcat 和 MySQL 可能互相传导),方向并不总是单向的。一刀切反转后,图里出现了 **A→B 又 B→A 的环**;代码自己的自检都警告 "graph may have cycles(图里可能有环)"。而后面找传播路径的算法默认图是没有环的(DAG),有环就会算乱。

**坑 5:把无关的 docker 节点也拉进图里凑数,增加噪声。**
dockerA1/A2/B1/B2 这些**没异常、根本不是嫌疑人**的节点也被拉进图,贡献一堆 0.1575 的噪声边,干扰判断。而且(见 3.6)它们的类型还被认错了。

## 3.3 InterventionAgent + CounterfactualAgent:打分和最终裁决

### 它怎么工作的
- InterventionAgent 用公式 `根因分 = 0.40×证据 + 0.25×是否像源头 + 0.20×时间早 + 0.15×解释分` 排名次。
- CounterfactualAgent 做**反事实(counterfactual)**裁决。

  > **反事实**,大白话:**做一个"假如"的思想实验——"假如这个嫌疑人不是真凶,现在这些异常还能不能被别人更好地解释?"** 如果别人也能解释,说明它可能只是中间传播的、不是源头,要扣分。

### 坏在哪
- **坑 1:这不是真的"干预/反事实",只是套了名字的加权公式。** 真正的因果学里,"干预"要做"假如把这台机器修好,系统会怎样"的重新推演;这里只是几个分数相加,没有任何重新推演。
- **坑 2:反事实分基本是个常数 0.50。** 13 个样本的反事实分全在 0.50 上下,**这一层等于没起作用**。
- **坑 3:置信度恒定在 0.5,导致"补救机制"永不启动。** 系统本来设计了"如果没把握就重试/换策略"的补救逻辑,但触发条件是"置信度低于阈值"。结果置信度永远卡在 0.5 这个不高不低的位置,**补救从来不触发**,错了也不会自我纠正。

## 3.4 时间定位:为什么时间总差几分钟

### 它怎么工作的
选定真凶后,从真凶的某一条异常曲线里找"**起点(onset)**"——

> **起点(onset)**,大白话:**这台机器从"正常"变成"出事"的那个拐点时刻。**

找起点的规则是:如果异常段一开头就接近最高峰,就取段的开头;否则找最大的那个跳变点;再不行取最高峰。

### 坏在哪
- **坑 1:因为前面"饱和",异常段往往一开头就是平顶(一上来就满格)**,所以规则永远命中第一条"取段开头",而段开头又被"窗口前扩5分钟"往前挪了,**系统性偏早**(样本1早16分钟)。
- **坑 2:时间只从"一个组件的一条曲线"上取**,没有**把多个相关信号、日志报错时刻、调用延迟突增**放在一起交叉验证。所以忽早忽晚,纯看运气(样本4反而晚了7分钟)。

## 3.5 多智能体 / 大模型:名不副实

- **大模型(LLM)全程是关着的**(`use_llm_reasoning=False`)。`prompts.py` 里写好了一堆精致的提示词,**但从来没被调用**,是死代码。
- **所谓"多智能体"其实是 7 个函数按固定顺序各跑一次**,共用一个数据字典传结果。调度逻辑是一串 `if-else`,不是真正的"思考/决策"。

> 大白话:**它叫"多智能体",但智能体之间不会互相质疑、不会争论、不会主动去查更多证据、也没有大模型参与推理。它更像一条死板的传送带,而不是一个会推理的团队。** 这和"LLM多智能体因果推理"的项目定位是矛盾的。

## 3.6 工程 bug 和写死的"魔法数字"

- **G1 组件类型认错**:`infer_component_type` 把 dockerA1/A2/B1/B2 错判成 "service"(其实是容器)。而且代码的分类和数据集里写的 `level`(record.csv 把 Mysql02 标成 pod)对不上。
- **G2 自我诊断坏了(对应现象5)**:EvaluationAgent 判"时间错"的前提是"组件得先猜对"。但问时间的任务里组件根本不打分、又往往猜错,于是"时间错"这条永远进不去,全掉进"未知错误 TOOL-ERROR"。**专门诊断时间错误的逻辑,对时间任务是瞎的。**
- **G3 标准答案对齐脆弱**:用"第几行对第几行"的方式把题目和答案配对,而题目文件因为有多行文本,行数和答案文件对不齐,稍有错位答案全错。
- **G4 满屏写死的"魔法数字"**:`0.65/0.35`、`0.40/0.25/0.20/0.15`、`0.55/0.25/0.20`、各种 `0.58/0.70/0.78` 的保留比例、整张类型先验表……**这些都是拍脑袋定的常数,不是从数据里学出来的**,既脆弱又不通用。讽刺的是,**项目自己就违背了它声称的"不靠写死规则"的原则。**
- **G5 原因复读机(对应现象3)**:原因打分靠关键词家族投票,而满地的 OS 内存共模信号让"内存家族"票数永远最高,于是不管真相是什么都输出"内存不足"。

# 四、把上面所有问题归纳成"几条根本病根"

把几十个具体问题往上提炼,其实是 **5 条根本病根**:

1. **病根 A:警报器坏了(异常检测饱和 + 抹平独家线索)。** 人人都报警,关键的独家报警反而被静音。→ 后面全步建立在噪声上。
2. **病根 B:关系图对真凶失明,且打分奖励"朋友圈大"而非"真闯祸"。** 数据库是孤岛看不见,网关朋友圈大被偏爱。→ 系统性指认 MG01/MG02。
3. **病根 C:分数和真相脱钩,全挤成常数。** 因果图退化成固定表,打分失去区分力,补救机制永不启动。→ 越分析越没信息。
4. **病根 D:没有真正的"找时间起点"和"问原因"的机制。** 时间靠瞎蒙、原因当复读机。
5. **病根 E:名为多智能体/大模型,实为死板流水线;连自我诊断都坏。** 没有推理、没有争论、没有纠错。

**一句话总病根:这套系统现在是一个"被网关枢纽和内存先验牵着走的、几乎输出固定答案的机器",真实的故障数据几乎没参与最终决策。** 这是机制层面的退化,**绝不是调参能救的**。

# 五、完整优化方案(每一步都讲透:现状→改成什么→为什么更好→怎么和多智能体结合)

下面 10 个改进点。每条都遵守你的要求:**只改机制和算法,不针对这个数据集写死阈值/规则**。所有"为什么更好"都讲成大白话,并标注参考了哪些成熟成果。

## 改进 1:让关系图"看得见"数据库——构建异构因果图(治病根 B)

- **现状:** 数据库/缓存不在调用记录里 → 在图里是孤岛 → 永远不被当真凶。
- **改成什么:** 不再只靠调用记录画图,而是**三种线索一起画**:
  1. **调用边**:从 trace 来(谁调用谁);
  2. **资源依赖边**:从配置/CMDB 来(哪个服务连哪个数据库),即使调用记录里没有;
  3. **统计因果边**:在指标数据上用因果发现算法(下条详述)直接算出"谁的波动带动了谁的波动",**把孤立的数据库重新接回图里**。
- **为什么更好(大白话):** 真凶数据库一旦有了连线,它"能影响下游"这件事才看得见,才有机会被打分选中。
- **怎么算"统计因果边":** 在去掉背景共模后的指标残差上,跑成熟的因果发现算法(PC 算法、Granger 因果、LiNGAM 等)。

  > **因果发现(causal discovery)**,大白话:**给一堆随时间变化的曲线,用统计方法自动推断"谁的变化引起了谁的变化",而不是只看"谁和谁一起变(相关)"。** 相关不等于因果(冰淇淋销量和溺水人数相关,但不是因果);因果发现就是要把这层区分开。
  > **条件独立性检验**,大白话:**控制住第三个变量后,看 A 和 B 还相不相关。如果一控制住 C,A 和 B 就不相关了,说明 A、B 不是直接因果,而是被 C 共同带动的。** 这是 PC 算法定方向的核心工具。

- **和多智能体结合:** 由一个 **"假设生成智能体"** 产出这张图,但**明确把它当作"初步假设"而非"铁的事实"**,交给后面的智能体去质疑、修正(这是 GALA 框架的核心思想:图只是假设种子)。
- **参考成果:** CausalRCA、CIRCA(KDD'22)、RCD —— 都是"用调用拓扑约束 + 在指标上做因果发现"的成熟范式。

## 改进 2:修好警报器——共模剔除 + 独家线索保护 + 概率化报警(治病根 A)

- **现状:** 人人报警(饱和),独家关键报警被抹平,配置常量被误报。
- **改成什么(三件套):**
  1. **去掉"死数"指标**:用方差/熵的检验,自动剔除像 `HeapMemoryMax` 这种几乎不变的常量。

     > 道理:**一条几乎不动的曲线本身不含任何信息,留着只会添乱。** 这是通用原理,不是针对本数据集。
  2. **剔除背景共模**:用稳健主成分分析(robust PCA)等方法,把"全机房一起涨跌"的背景波动减掉,**只留下每台机器自己特有的、减完背景后的残差**。

     > 大白话:**先把"全班都考差(卷子难)"这部分扣掉,剩下的才是"某个同学特别差"的真信号。** 这一招直接根治"人人报警"的饱和问题。
  3. **保护独家线索(关键修复)**:不再用"除以所有组件中位数"的归一化(那会把独家信号抹平),改用**逆频率加权**——**越是"只有少数组件才有"的指标,权重越高**。

     > 大白话:**"全机房都有的指标"信息量低,"只有数据库才有的报警"信息量高,应该更被重视。** 这样 Mysql02 的 `Aborted Clients=5.6` 才能脱颖而出。这个思想来自信息检索里的 IDF(逆文档频率),是通用方法。
- **再加一层"概率化报警"**:用极值理论(EVT/SPOT)给出**"这个偏差有多反常"的概率**,而不是简单的"超没超线"的 0/1。

  > **极值理论**,大白话:**专门研究"罕见的极端值"的统计学。** 它能动态算出"超过多少才算真的反常",比固定取 P95 更稳、更不容易误报。
- **参考成果:** BARO、SPOT(KDD'17)、谱残差;IDF 来自信息检索。

## 改进 3:用真正的"因果干预/反事实"打分——别再奖励朋友圈(治病根 B、C)

- **现状:** 解释分奖励"连线多"(朋友圈),反事实分是常数。
- **改成什么:** 在因果图上**拟合一个结构因果模型(SCM)**,然后对每个嫌疑人做真正的因果检验:

  > **结构因果模型(SCM)**,大白话:**给每个组件建一个小公式,描述"它的指标是怎么由它的上游(父节点)决定的"。** 有了这套公式,就能做"假如"推演。
  > **干预(do-operator)**,大白话:**"假如我强行把这台机器恢复正常,整个系统会变成什么样?"** 把这台机器设为正常后重新推演,如果**大部分异常都消失了**,说明它很可能是真凶;如果异常还在,说明它不是。

- **怎么打分(关键转变):** 不再看"你朋友圈多大",而是看 **"你这台机器的异常,有多少是它的上游解释不了的(它自己的'冤情'有多独特)"**,以及 **"假如修好它,能消解多少下游异常"**。
- **为什么更好(大白话):** 数据库是孤岛、朋友圈小,但它的异常**没有任何上游能解释**(它就是源头),按新打分它会得高分;网关 MG02 的异常其实**能被它的上游解释**(它只是中间传话的),按新打分会被扣分。**这样就把"奖励朋友圈"扭转成"奖励真源头",直接根治老指认 MG02 的毛病。**
- **理论依据:** Pearl 因果阶梯(关联→干预→反事实,逐级更接近真因果)、CIRCA 的回归不变量检验、Halpern-Pearl 的"实际因果(actual causality)"理论。
- **和多智能体结合:** 由 **"反事实质疑智能体"** 专门对每个假设做"假如它不是真凶"的反驳,产出反驳证据。

## 改进 4:用"系统级找起点"定位时间——多信号一起投票(治病根 D)

- **现状:** 时间只从一个组件一条曲线取,且偏早。
- **改成什么:**
  1. 在残差信号上跑**变点检测**算法(BOCPD、CUSUM、e-divisive),找每条信号"由稳定变突变"的拐点;

     > **变点检测(change-point detection)**,大白话:**自动找出一条曲线"性质突然改变"的那个时刻**(比如从平稳突然开始飙升)。
  2. **沿因果源的祖先链**,取**最早的那个显著拐点**,并要求**多个相关信号互相印证**;
  3. 再用**日志里的报错爆发时刻**和**调用延迟的突增时刻**做**三角验证**。
- **为什么更好(大白话):** **不再靠一台机器一条线瞎蒙,而是让"多条曲线 + 日志 + 调用延迟"一起投票,谁的拐点最早又最一致,就定谁。** 这样时间才稳。
- **和多智能体结合:** 由 **"时间推理智能体"** 综合多源证据仲裁出最终时间。

## 改进 5:让"多智能体"名副其实——假设、辩论、验证的闭环(治病根 E)

- **现状:** 7 个函数死板串行,大模型关着。
- **改成什么:** 把角色重构成**会互相质疑、会查证据的真智能体团队**:
  - **假设智能体**:产出带证据的嫌疑人名单(基于改进1、2);
  - **反事实质疑智能体**:逐一反驳(改进3);
  - **验证智能体**:用"分模态证据包"交叉核对——

    > **分模态证据包**,大白话:**不把指标、日志、调用链揉成一锅,而是分门别类整理好**(一份指标趋势图、一份调用关系子图、一份过滤后的报错日志摘要),让大模型分别看,减少误判。这是 GALA、TAMO 的做法。
  - **裁决智能体**:综合各方意见排名;**如果把握不够,就主动要求"再深入查一查"**,而不是草草定案;
  - **规划智能体**:见改进6。
- **为什么更好(大白话):** **真凶不再由一条死板流水线一锤定音,而是由一个会争论、会补查证据、会反悔的团队讨论出来。** 大模型在这里扮演"提出领域常识 + 批评不合理假设"的角色,而不是直接拍板(避免它瞎编)。
- **参考成果:** GALA(图增强的大模型智能体)、Flow-of-Action(WWW'25,用标准作业流程 SOP 约束大模型、减少幻觉)、TAMO/RCAgent(工具辅助)。

## 改进 6:按"题目要什么"来安排流程(治病根的架构错配)

- **现状:** 不管题目问时间、问组件还是问原因,都跑一整套组件流程。
- **改成什么:** 加一个 **"规划智能体"**,先读题目要求(`target_fields` 字段里本来就写明了要什么):
  - 只问时间 → 主要跑改进4的找起点流程;
  - 只问组件 → 主要跑改进1+3;
  - 问原因 → 跑改进10的原因归因。
- **注意合规:** 这是**读题目本身的明确要求**(字段早就存在),**不是看标准答案反推**,完全合规。还能省掉无关步骤(现在为一道只问时间的题白白跑 166 秒)。
- **参考成果:** Flow-of-Action 的 SOP 编排、ReAct 规划。

## 改进 7:让分数恢复区分力 + 让系统知道自己有没有错(治病根 C、E)

- **现状:** 所有分数挤成常数(0.5~0.7),补救从不触发,自我诊断坏了。
- **改成什么:**
  1. **打分校准**:引入**对比式排序**和**温度校准**,并加一条约束:**不同的真凶必须算出明显不同的分数**(用分布散度度量),从机制上打破"分数全挤成一团"。

     > **校准(calibration)**,大白话:**让"模型说的把握"和"实际答对的概率"对上**。模型说 80% 把握,就该有 80% 真答对。现在它永远说 50%,既没用也不准。
  2. **修好补救触发**:改成"当因果解释不充分时"才触发重查,而不是看那个永远 0.5 的置信度。
  3. **修好自我诊断**:让时间任务即使组件猜错也能正确判成"时间错(CF-TIME)";顺手修好组件类型认错(G1)、答案对齐(G3)。
- **为什么更好:** 系统终于能**对不同样本给出不同的、可信的把握度**,并在没把握时**主动重查、纠错**。

## 改进 8:顶会级的评测方法(决定能不能发 ICSE/FSE/AAAI)

- **现状:** 只有"几个样本得分低"这种零散结果,够不上任何顶会的实证门槛。
- **改成什么:**
  1. **对比基线**:和 CIRCA、RCD、BARO、CausalRCA、纯 GPT-4o,以及**"直接挑证据最强组件"的朴素基线**比。

     > 关键:**要用朴素基线证明"现有这套因果机制还不如朴素基线"(也就是现象2/第0部分),再证明"改进后明显超过朴素基线和现有 SOTA"。**
  2. **跨数据集泛化**:在 Bank/Telecom/Market 以及公开基准(RCAEval、PetShop)上测;用**"留一种故障类型/留一个系统"**的方式验证泛化,杜绝过拟合。
  3. **分阶段指标**:异常检测准确率、因果图结构准确率(SHD、定向正确率)、组件 Top-k 命中、时间误差(分钟)、原因准确率、**校准误差(ECE)**。
  4. **统计显著性**:多随机种子 + Wilcoxon 检验 + 效应量,而不是单次跑分。
  5. **消融实验**:逐个关掉(去共模、去统计因果边、去反事实、去大模型辩论、去校准),证明每个模块都有用。
  6. **去魔法数字**:所有权重尽量由模型拟合/数据自适应得出,并做敏感性分析,**正面回应"不许网格搜索迎合测试集"**。

## 改进 9:打分校准与"可辨识性"专项(把现象1当成一个要解决的科学问题)

- **这是把"现象1:分数全挤成一团"单独提炼成一个研究贡献。**

  > **可辨识(identifiable)**,大白话:**不同的真相能产生不同的、可区分的输出。** 现在系统"换真凶分数也不变",叫"不可辨识";改进目标是让"真凶不同 → 分数明显不同"。
- **做法:** 用**反事实可辨识性检验**保证"改一个根因,最终分数会随之明显变化",并把这个性质作为论文的理论卖点(AAAI 看重)。

## 改进 10:用"反事实"问原因,告别复读机(治病根 D 的原因部分)

- **现状:** 原因靠关键词家族投票,被内存信号带偏,永远答"内存不足"。
- **改成什么:** 对每个"嫌疑组件—候选原因"的组合做**反事实归因**:"假如这个原因(比如网络丢包)恢复正常,观测到的异常能不能消解?" **选那个最能消解异常的原因**,而不是数关键词。
- **为什么更好(大白话):** **从"哪个关键词出现最多"变成"哪个原因最能解释眼前的异常",内存先验就压不住网络/磁盘的真原因了。**
- **和多智能体结合:** 反事实质疑智能体顺带产出原因归因。

# 八、名词小词典(随手查)

- **根因定位**:从一堆报警里找出真正的罪魁祸首(组件/时间/原因)。
- **多智能体**:把任务拆给多个小角色,各管一摊,协作完成。
- **阈值 / P95**:判断"正常 vs 异常"的标准线;P95 = 一整天数值排序后 95% 位置的那个值。
- **偏差(deviation)**:实际值超过标准线多少。
- **饱和**:警报太灵敏,人人都在报,等于没报。
- **共模信号**:整个机房一起涨跌的背景波动,不是某一台的故障。
- **独家线索 / IDF 加权**:只有少数组件才有的指标信息量大,应更被重视。
- **调用图**:谁调用了谁。**因果图**:谁出问题会连累谁。
- **边权重**:一条影响关系有多强。
- **ExplainScore(解释分)**:一个嫌疑人能顺着连线解释掉多少别人的异常(现缺陷:偏爱朋友圈大的)。
- **相关 vs 因果**:一起变 ≠ 一个引起另一个;因果发现就是把这层区分开。
- **因果发现 / PC算法 / Granger**:用统计方法自动推断"谁引起谁"。
- **条件独立性检验**:控制住第三方后看俩还相不相关,用来辨别真因果。
- **SCM(结构因果模型)**:给每个组件建小公式,描述它怎么被上游决定,从而能做"假如"推演。
- **干预(do)**:"假如把这台修好,系统会怎样"的重新推演。
- **反事实**:"假如它不是真凶/这个原因不存在,异常还在不在"的思想实验。
- **Pearl因果阶梯**:关联 → 干预 → 反事实,逐级更接近真因果。
- **起点(onset)/变点检测**:一台机器从正常变出事的拐点时刻 / 自动找拐点的算法。
- **校准 / ECE**:让"模型说的把握"和"实际答对率"对上;ECE 是衡量两者差距的指标。
- **可辨识**:不同真相能产生不同、可区分的输出(现缺陷:换真凶分数也不变)。
- **极值理论(EVT/SPOT)**:专门研究极端罕见值的统计学,用来更稳地定"多反常才算异常"。
- **分模态证据包**:把指标/日志/调用链分门别类整理给大模型看,减少误判。
- **SOP**:标准作业流程,用来约束大模型按步骤走、少瞎编。
